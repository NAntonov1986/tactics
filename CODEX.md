## Краткий обзор режима

PvE: игрок управляет наёмным отрядом (целевой размер — около 8 героев). На каждую миссию выбирается до 4 героев из партии, остальные ждут. Выбранные размещаются на 4 стартовых клетках (ряд 19, столбцы 8/9/10/11), монстры спавнятся в случайных точках верхней половины карты (ряды 0–9).

**Стартовое состояние (Camp v2-economy, 13.05.2026).** Партия в начале нового забега **пустая**. Игрок получает стартовый капитал `ECONOMY.START_CAPITAL` (2 000 g по дефолту — на 8 героев 1-го уровня (1 600 g) + ≈400 g запас на разброс цен и расходники) и нанимает первых наёмников через палатку «Найм» в лагере. Онбординг начинается с формирования отряда из пула в 6 случайных кандидатов 1-го уровня (равная вероятность 4 классов: warrior/archer/mage/priest). Ранее (Camp v1.5-priest, 09.05.2026) выдавалось 8 готовых героев бесплатно (2 каждого из 4 классов); этот режим заменён на «капитал + найм» в Camp v2-economy.

**Регионы и прогрессия (Camp v1.5).** Мир разбит на 5 регионов (`r1..r5`), каждый со своей текущей сложностью (стартово 1). UI глобальной карты (Camp v1.5-UI) показывает регионы как карточки с цветовой индикацией: 1-2 зелёный («Лёгкий»), 3-5 жёлтый («Средний»), 6-9 оранжевый («Тяжёлый»), 10+ красный («Смертельный»). После каждой ВЫИГРАННОЙ миссии: целевой регион -3 (clamp 1), все остальные +1 → net +1 к суммарной сложности мира за миссию (X-COM-style давление). После ПРОВАЛЕННОЙ миссии: ВСЕ регионы +1. Сложность миссии = `region.difficulty` (используется и для спавна монстров, и для генератора награды).

**Выбор отряда (Camp v1.5-UI).** Клик «Выбрать отряд» на регионе ведёт на экран `missionSetup`: карточки всех героев партии (портрет, уровень, HP/мана, иконки экипировки и скиллов). Игрок отмечает 1-4 героев (лимит = `HERO_SPAWN.length` = 4 стартовых клетки; отдыхающие/мёртвые недоступны), нажимает «Отправить отряд» — стартует `startMission(regionId, heroIds)`. Кнопка «Отмена» возвращает на глобальную карту без последствий.

**Permadeath.** Мёртвые в миссии герои навсегда удаляются из `state.party` (после level-up очереди в `endMissionCleanup`). Если в партии не осталось живых героев → `state.gameOver='defeat'`.

**Отдых.** Герои, ходившие на миссию, получают `restingTurnsLeft = 1` и не могут идти на следующую миссию (тикается в начале каждой пост-миссионной очистки). Это вынуждает чередовать состав.

Цель миссии — уничтожить всех монстров. При успехе выжившие участники получают +1 уровень (level-up очередь, окно прокачки), затем партия возвращается в лагерь.

Все герои в партии мертвы — экран финального поражения с кнопкой «Новый забег».

## Характеристики и производные значения

Семь базовых характеристик у всех юнитов:

| Ключ | Название | Роль |
|---|---|---|
| `str` | Сила | Урон оружия на силе (меч, когти) |
| `dex` | Ловкость | Урон лука лучника |
| `vit` | Живучесть | `maxHp = baseHp + Живучесть × 2` |
| `spd` | Скорость | Инициатива и `moveRange = 1 + ⌊Скорость / 2⌋` |
| `wis` | Мудрость | Урон жезла мага и магических навыков |
| `int` | Интеллект | `maxMana = 5 + Интеллект × 2` |
| `luk` | Удача | Шанс крита (%) и тай-брейк инициативы |

Все деления — с округлением вниз.

Производные:
- `maxHp(baseHp, vit) = baseHp + vit × 2`
- `maxMana(int) = 5 + int × 2`
- `moveRange(spd) = 1 + ⌊(√(8·spd + 1) − 1) / 2⌋` — треугольная цена: каждая k-я доп. клетка стоит k единиц Spd (накопительно). Таблица: spd 0 → 1, 1–2 → 2, 3–5 → 3, 6–9 → 4, 10–14 → 5, 15–20 → 6, 21–27 → 7, 28–35 → 8, 36–44 → 9. Каскадно влияет на дальность рывковых навыков воина (charge/heroic_charge) и скиллов с `castRangeMul`.
- `critChance(luk) = luk` (%); может быть модифицирован пассивками — см. [Шанс крита](#шанс-крита).
- `damage(weapon) = weapon.formula.base + ⌊stats[weapon.formula.stat] / weapon.formula.divisor⌋`

**Снижение максимумов во время боя.** Если `vit`/`int` уменьшаются (дебафф) и новый `maxHp`/`maxMana` падает ниже текущего HP/маны — текущее значение **обрезается** до нового максимума. Может убить юнита (HP до 0). При увеличении максимума текущее значение НЕ растёт автоматически.

**Эффективные статы.** Все формулы боя/AI читают **эффективные** статы (база + сумма `statMods` всех висящих эффектов, clamp ≥ 0) через `effectiveStats(unit)` в `core/stats-calc.js`. Исключение — `weaponRangeBonus`: специфический модификатор оружейной дистанции, в `effectiveStats` не входит, читается отдельно через `weaponRangeOf(unit)` в `data/weapons.js`.

## Уровни

У каждого юнита есть `level`. Отображается на панели рядом с именем класса.
- **У героев** на старте `level = 0` (без скиллов). Перед первой волной запускается стартовый level-up 0→1, в котором каждый герой выбирает первый скилл. Дальше +1 за каждую пройденную волну (см. [Прокачка героев](#прокачка-героев)).
- **У монстров** `level` (personalLevel) = `max(1, wave.number - cls.baseDifficulty + 1)`. У зомби `baseDifficulty=1`, поэтому его уровень = номеру волны. У монстров с большим `baseDifficulty` (когда появятся, например, голем с baseDifficulty=20) — уровень начинается с 1 на «своей» волне и растёт линейно. Подробнее — [Метаданные монстра](#метаданные-монстра). Прокачка статов и тира пассивок описана в [Зомби](#зомби).

## Прокачка героев

**Стартовый level-up.** На старте новой игры (или `restartGame`) герои создаются с `level=0` и пустыми `activeSkillsOverride/passiveSkillsOverride`. Первая `startNextWave` вместо спавна зомби запускает очередь level-up 0→1 — каждый герой выбирает первый навык. После завершения очереди — обычная первая волна.

**Между волнами.** После победы в текущей волне (до спавна следующей) каждый выживший герой получает один уровень. По очереди для каждого героя выскакивает модальное окно прокачки (`render-level-up.js` → `state.activeLevelUp` + `state.levelUpQueue`); пока окно открыто, поле боя недоступно.

### Шаги одного апа

1. **Авто-рост +1.** `applyLevelBump(unit)` (в `core/level-up.js`) инкрементирует `unit.level` и кидает ролл по таблице класса:
   - **70%** — случайная характеристика из `mainStats`;
   - **30%** — случайная из `secondaryStats`.
   Прочие статы (не main и не secondary) при ролле **никогда** не выпадают. Имя выпавшего стата отображается в шапке окна апа («Авто-рост: Сила +1»).

2. **Выбор игрока — зависит от kind, который вычисляет `getLevelUpKind(unit, level)`:**

   - **`stats`** (чётные уровни 2, 4, 6, …): показ 7 иконок характеристик, выбор +2 (`applyStatChoice`). Можно поднять любую, не только main/secondary.
   - **`skills-new`** (нечётный, известных <5): показ **до 4 случайных** ещё не изученных скиллов из `skillPool`. Если активные слоты 4/4 — пул только из пассивов, и наоборот. Если в пуле меньше 4 — карточек столько, сколько есть. Выбранный кладётся в первый свободный слот (active/passive по `SKILLS[id].kind`), тир `basic` (`learnNewSkill`).
   - **`skills-upgrade-basic`** (нечётный, известных = 5, есть basic): показ **до 4 случайных** скиллов с тиром `basic`. Выбор повышает basic→advanced. Эта фаза держится, пока хоть один навык в basic — advanced→elite не предлагается, пока все базовые не апнуты.
   - **`skills-upgrade-advanced`** (нечётный, все basic уже advanced, есть advanced): показ **до 4 случайных** скиллов с тиром `advanced`. Выбор повышает advanced→elite.
   - **`stat-bonus`** (нечётный, все 5 скиллов в elite — терминальная фаза): показ 7 иконок статов как в `stats`. Подзаголовок: «Все навыки в Элитном тире. Бонус: +2 к характеристике». Это значит, что после full-elite КАЖДЫЙ нечётный уровень даёт +2 (а не +1, как авто-рост) — компенсация за исчерпанные скилл-улучшения.

3. После выбора — `finishCurrentLevelUp` сдвигает очередь, открывается окно следующего героя. Когда очередь пуста — `startNextWave` (с обычной задержкой 900 мс).

### Максимум навыков

`MAX_SKILLS = 5` (в `core/level-up.js`). Любое распределение от 4 активных + 1 пассивный до 1 активный + 4 пассивных — допустимо. Слотов в инстансе по 4 на каждую категорию, но новые скиллы не выдаются после набранных 5, поэтому 4А+4П невозможно.

### Таблицы статов по классам

| Класс | mainStats (70%) | secondaryStats (30%) | Прочие (0% автоматически) |
|---|---|---|---|
| Воин | str, vit | dex, spd, luk | wis, int |
| Лучник | dex, spd | vit, luk, str | wis, int |
| Маг | wis, int | luk, vit, spd | str, dex |

Источник правды — `CLASSES[classId].mainStats / secondaryStats` в `data/classes.js`.

### Мёртвые герои

Герой, погибший в волне, не прокачивается (его id не попадает в `state.levelUpQueue`). Воскрешения нет — он остаётся мёртвым между волнами, как и раньше.

### Запуск победы извне

Логика «волна пройдена» выделена в `forceWaveVictory()` (в `core/state.js`). Вызывается:
- из `checkVictory` — при отсутствии живых монстров (классическое условие);
- из DevTools — кнопкой «Победа волны» (зелёная) для отладки прокачки без зачистки карты.

Это задел под будущие типы миссий с другими условиями победы (защита каравана, выживание N раундов и т.п.) — единая точка перехода в победный исход.

## Типы юнитов

Каждый класс помечен одним из типов из реестра `UNIT_TYPES`. Тип влияет на восприятие урона по типу.

| Ключ | Название | Кто относится |
|---|---|---|
| `living` | живое существо | Воин, Лучник, Маг |
| `mechanism` | механизм | — |
| `undead` | нежить | Зомби |
| `demon` | демон | — |
| `elemental` | элементаль | — |

Действующие правила (все считаются в `computeIncomingDamage` в `core/damage.js`):

| Тип цели | Поведение |
|---|---|
| `mechanism`, `undead`, `elemental` | Иммунитет к **ядовитому** урону (`damageType: 'poison'` → 0). Эффекты «Отравлен» и «Трупный яд» на цели не накладываются. |
| `undead`, `demon` | Уязвимость к **святому**: `damageType: 'holy'` × 1.5 (округление вниз, ПОСЛЕ крита). |

Чтобы добавить новое правило по типу — одна ветка в `computeIncomingDamage`, все исполнители подхватят автоматически.

## Метаданные монстра

Каждый класс с `kind: 'monster'` имеет три метаполя поверх обычной классовой структуры (правка 06.05.2026, задел под систему миссий и состав волн):

| Поле | Тип | Назначение |
|---|---|---|
| `unitType` | строка из `UNIT_TYPES` | Боевая механика: иммунитеты по типу (poison у undead/mechanism/elemental), уязвимости (holy у undead/demon). Уже используется в `core/damage.js`. |
| `group` | строка | Фракционная принадлежность для будущей логики состава волн и взаимодействий «свой / чужой». У зомби — `'undead'`. У будущих волков, например, — `'wolves'`. Сейчас просто хранится. |
| `baseDifficulty` | число ≥ 1 | Минимальный номер волны, на которой монстр может появиться. Personal level считается как `max(1, wave.number - baseDifficulty + 1)` в `spawnGroupWave` (см. `core/state.js`). У зомби и волка — 1 (могут появляться с 1-й волны), у Волка-вожака — 5. |
| `isLeader` | bool | `true` для мини-боссов / вожаков своей группы. Сейчас просто метаданные — задел под состав волн (один лидер на N рядовых) и UI-индикатор. |

**Идея baseDifficulty.** Монстр, появляющийся позже (например, голем с baseDifficulty=20), на своей «первой» волне имеет personalLevel=1 — но **базовые статы** у него существенно выше зомби. То есть голем lvl 1 на 20-й волне должен быть как минимум не слабее зомби lvl 20. Это балансируется в самих `CLASSES[id].stats` и `CLASS_PROGRESSIONS[id]` нового монстра — фреймворк только обеспечивает, что монстр **не появится** раньше своей baseDifficulty (это будущая логика волн будет уважать).

**Где править формулу personal level:** `spawnGroupWave` в `core/state.js`. Реестр групп — `WAVE_GROUPS` (рядом). `spawnZombieWave` оставлена для DevTools/тестов и в startNextWave не используется.

## Навыки

Каждый навык — активный или пассивный. У большинства три тира: **Базовый**, **Продвинутый**, **Элитный**. Тир хранится на юните отдельно от id навыка (`unit.skills[i] = { id, tier }`); параметры читаются из `SKILLS[id].tiers[tier]`. Поля общие для всех тиров — на верхнем уровне записи скилла.

**Источник правды по конкретным навыкам — `src/data/skills.js`.** У каждой записи есть подробный комментарий-шапка с описанием поведения, тиров и правил. Дублировать здесь конкретику было бы постоянным источником рассинхрона.

**Сводный справочник по тирам** — автогенерируемый markdown `docs/skills.md`. Собирается скриптом `tools/gen-skills.js`: загружает `data/skills.js` и зависимости, использует `render/skill-tooltip.js` для построения цифр по каждому тиру, пишет результат в `docs/skills.md`. Удобно для размышлений над балансом — все цифры в одном файле. Запуск из корня проекта: `node tools/gen-skills.js`. Запускать после балансных правок (или как pre-commit hook, если оформишь). Файл `docs/skills.md` редактировать вручную НЕ нужно — будет перезаписан.

### Конвенции

- `kind`: `'active'`, `'passive'`, или `'effect'` (запись висящего эффекта/баффа, без активации).
- `delivery` (для активов): `'melee'` / `'ranged'` / `'aoe'` / `'self_buff'` / `'self_aura'` / `'cleanse'` / `'teleport'` / `'leap'`.
- `range: 0` для self-only активов (mana_focus, shield_block, second_attack, и т.п.).
- `consumesBasicAttack: true/false` — расходует ли каст активного скилла обычную атаку юнита. По умолчанию false для self-buff.
- `canCrit: true/false` — может ли скилл критовать (DoT-тики и self-buff обычно нет).
- `manaCost`, `cooldown` — задаются ПО ТИРУ (в `tiers[tier]`), не на верхнем уровне.
- Унифицированные кулдауны активок воина и лучника = **5** (исключение — `cover` = 3).

## Эффекты (баффы/дебаффы)

Эффект на юните — это запись в `unit.effects: [{ id, ...поля }]`. Поля специфичны для эффекта, но соблюдаются общие конвенции.

### Жизненный цикл эффекта

| Поле «истечения» | Когда снимается | Где обрабатывается |
|---|---|---|
| `remaining: N` | В конце своего хода носителя `remaining -= 1`; снимается при достижении 0. | `core/effects.js → tickEffectsAtTurnEnd`. |
| `expiresAt: 'turnStart'` | В начале СЛЕДУЮЩЕГО собственного хода носителя, ДО `onTurnStart`-фазы. | `core/effects.js → expireTurnStartEffects`. |
| `expiresAt: 'turnEnd'` | В конце ТЕКУЩЕГО хода носителя, ПОСЛЕ всех остальных фаз (живёт ровно один свой ход). | `core/effects.js → expireTurnEndEffects`. |
| `expiresAt: 'nextAttack'` | После первой же атаки носителя, через `consumeNextAttackEffects` в `executeAttack`. | `core/effects.js → consumeNextAttackEffects`. |
| `expiresAt: 'forcedMove'` | После того как носитель совершил «вынужденное действие» (атака или шаг к forcedTarget). | `core/effects.js → consumeForcedMoveEffects` (зовётся из ai.js). |
| `charges: N` | Расходуется в фазе 2 `computeIncomingDamage` (armored: 1 заряд за единицу поглощённого урона). При `charges <= 0` удаляется. | `core/damage.js → computeIncomingDamage`. |
| `stacks: N` | Не тикает по времени. Сбрасывается отдельным механизмом (например, `expiresAt:'turnStart'` для `reinforcement`). | Зависит от эффекта. |

### Хуки по фазам носителя

Эффект может объявить хуки `onTurnStart(unit, eff)` и `onTurnEnd(unit, eff)` (на самой записи в `SKILLS` или на инстансе). Срабатывают ровно по одному разу на свой ход носителя, ДО уменьшения `remaining`.

Поиск хука: сначала на инстансе эффекта, затем `SKILLS[eff.id]`, затем `SKILLS[eff.id].tiers[eff.tier]` (тир-зависимый задел).

**Смерть носителя на хуке.** Если `onTurnStart` убил носителя — `beginTurn` через 280 мс уходит в `endTurn` (анимация смерти доигрывает). Если `onTurnEnd` убил — tick на трупе безвреден, инициатива идёт дальше.

**Эффекты, добавленные во время фазы**, в этой же фазе не сработают (итерация по снимку `unit.effects`).

### Стакинг

Базовое правило: **повторное наложение того же эффекта НЕ увеличивает силу**. Длительность увеличивается на новую длительность. Конкретные эффекты могут отступать от правила.

- `applyArmored`: суммирует заряды (`existing.charges += new`). Правка 07.05.2026: раньше был `max(existing, new)`, упиралось в потолок тира на повторных кастах. Кулдаун 5 ходов ограничивает частоту.
- `corpse_poison` тиров: высокий тир не заменяется низким, длительность продлевается.
- `reinforcement` стакается по `gainPerHit` за каждое попадание; снимается одним `expiresAt:'turnStart'`.
- DoT (`burning`, `poisoned`): `existing.remaining += новая длительность` — увеличивает урон следующего тика (урон тика = `remaining`).

**Снижение `vit`/`int` через дебафф** уменьшает `maxHp`/`maxMana`; если текущее значение выше нового максимума — обрезается. Может убить.

| id | Название | Иконка | Тип | Что делает |
|---|---|---|---|---|
| `burning` | Горит | 🔥 | огненный DoT | На `onTurnStart` носителя наносит `remaining` огненного урона (через `applyDamage(unit, dmg, null)`). |

| `stunned` | Оглушён | 💫 | контроль | На `onTurnStart` выставляет флаг `skipTurnThisTurn`, beginTurn моментально уходит в endTurn. Юнит не действует за этот ход. |
| `immobilized` | Обездвижен | 🪤 | контроль | Хука по фазам нет. Пока висит — `canUnitMove` возвращает false: блокируется режим движения, шаги ИИ, любой активный скилл с `movesUser: true`. Атака и self-buff остаются. |
| `camouflage` | Маскировка | 🍃 | скрытность | Хука по фазам нет. Иммунитет: `unitType === 'mechanism'` (фильтрует `applyFrightened`, эффект не накладывается). DoT не критуют (урон = `remaining`, без ролла). Apply-хелперы: `applyBurning`, `applyPoisoned`, `applyStunned`, `applyImmobilized`, `applyCamouflage(unit, duration?)`, `applyFrightened(unit, duration)` — все в `core/effects.js`. `removeCamouflage(unit, reason)` — точечное снятие при разоблачающих событиях (атака/движение/любой активный навык), вызывается безусловно из `executeAttack`/`executeMove`/`onActiveSkillCast` (no-op если эффекта нет).

**Источник применения статус-эффекта при касте навыка** — поле тира `applyEffect: { id, duration, strength?, chance? }`. Диспатчер — `core/skills.js → applySkillEffectDef`. AoE-навыки применяют после урона, на выживших.

## Бой: общие правила

- Сетка: **20 × 20** клеток.
- Движение только ортогональное (диагонали запрещены). Живые юниты и надгробия блокируют движение; стрельба (манхэттен-дальность) проходит «сквозь».
- В режиме движения при наведении курсора показывается **превью маршрута** (`computeMovePath`): стрелки на промежуточных клетках, точка на финальной. Совпадает с фактическим путём.
- За свой ход юнит может: одно движение + одну атаку в любом порядке + **один активный навык** (общее правило: один актив за ход, неважно какой). Движение нельзя прервать.
- Павший юнит оставляет **надгробие** на своей клетке (герои и зомби одинаково). Надгробие блокирует движение, но не стрельбу. На переходе к новой волне надгробия очищаются.
- Инициатива: Скорость ↓, Удача ↓, случайный `unit.initiativeTiebreak` ↑ (фиксирован на бой). Считается по эффективным статам — дебаффы переставляют очередь в раунде.
- **Глобальное восполнение маны.** В начале своего хода каждый юнит получает `+1 маны` (`u.mana = min(maxMana, u.mana + 1)`), если `maxMana > 0` и не на максимуме. Это правило игры, не пассивный навык; пассивка `mana_regen` накладывается поверх.

## Расчёт урона: 3 фазы

`computeIncomingDamage(target, dmg, damageType, opts?)` в `core/damage.js`:

1. **Безусловные снижения** (без ресурса). Порядок применения: 1.1 `fire_shield` → `−damageReduction` для fire/frost (минимум 1). 1.2 `shield_block` → `−damageReduction` для любого типа (минимум 1). 1.3 `reinforcement` → `−stacks` для всех типов (минимум 1). 1.4 `endurance` → `−reduction` ТОЛЬКО для DoT-тиков (`opts.isDoTTick === true`); может обнулить полностью. 1.5 **`incomingReduction` от рясы священника** (балансная правка 14.05.2026, заменила прежнюю физ-фазу `armorFlat`) — `−N` ко ВСЕМУ входящему урону, ВКЛЮЧАЯ DoT-тики. Все типы (физика, магия, огонь, мороз, яд, святой) снижаются равнозначно. Минимум 1. Источник числа — `equipmentSpecialSum(target, 'incomingReduction')`. 1.6 **`bony`** (09.05.2026) — пассивка скелетов, `−reduction%` для атак с `delivery:'ranged'`, минимум 1. Не действует на melee/aoe/special. 1.7 **`evil_slayer`** (10.05.2026) — пассивка священника, `−reductionPercent%` от атак, чей `opts.source` имеет `unitType: 'undead'` или `'demon'`. Применяется ПОСЛЕ 1.5/1.6. 1.8 **`ghostly`** (12.05.2026) — пассивка Призрака, `−reduction%` от физического урона (минимум 1). Магия/огонь/мороз/электричество/святой/яд игнорируют ghostly. Все процентные ветки 1.6–1.8 мультипликативны; порядок между ними не влияет на итог.
2. **Расходуемые щиты.** `armored` → 1 заряд за 1 единицу поглощённого. При `charges <= 0` эффект удаляется. Полное поглощение обнуляет урон (пол «минимум 1» НЕ применяется на этой фазе).
3. **Резистенции/иммунитеты по типу.** Иммунитет к яду (`mechanism`/`undead`/`elemental`) → 0. Святой ×1.5 для `undead`/`demon`. Иммунитет = ровно 0 (не 1).

`damageType: 'special'` — short-circuit. Игнорирует все три фазы, возвращает `max(1, raw)`.

Фаза 2 МУТИРУЕТ эффект `armored` (расход charges) — осознанное отступление от pure-функций.

**Edge-case (подтверждено пользователем):** если фаза 3 потом обнулит урон полностью, фаза 2 всё равно тратит заряды. «Доспеху не важно, что под ним иммунно».

`applyDamage(target, dmg, source)` — единая точка снятия HP. До отнимания: `wakeOnDamage(target, source)` будит спящего NPC (если `dmg > 0`, source — живой враг); `triggerOnTakeDamagePassives(target, source)`. После: `triggerOnDealDamagePassives(source, target)` (если source — живой враг).

## Кулдауны

`unit.cooldowns: { [skillId]: turnsLeft }`. После успешного каста с `tierData.cooldown > 0` — `applyCooldown` ставит число; на каждом своём `endTurn` — `tickCooldowns` уменьшает на 1. При 0 — ключ удаляется. UI рисует слот серым с тултипом «(на откате: N ход(ов))».

На переходе к новой волне `cooldowns = {}` для всех живых юнитов (как и `skillsUsedThisTurn` / `actionsUsedThisTurn`).

## Шанс крита

`critChanceOf(unit)` в `core/stats-calc.js`:

1. Стартует с `effectiveStats(unit).luk`.
2. Если у юнита есть пассивка `crushing_magic` → `+ ⌊Мудрость × mult⌋` (по тиру: 1.0 / 1.5 / 2.0).
3. Если у юнита есть пассивка `marksman` → `+ bonus` (по тиру: 5 / 10 / 15).
4. Финальный clamp в `[-100, 100]`. 100 — гарантированный крит, −100 — гарантированный промах, 0 — обычный удар.

Крит удваивает урон. Один ролл крита на каст для AoE-навыков (удвоение применяется ко всей области).

## Объекты на поле

`state.objects: [{ id, kind, row, col, ownerTeam, payload }]`. Один объект на клетку (правило: `executeXxx` для размещения отказывает, если на целевой клетке уже есть объект). Юнит и объект могут находиться вместе — объект НЕ блокирует движение в общем случае; конкретные правила срабатывания — у каждого объекта.

Срабатывание объектов делегировано в `core/movement.js`:
- `triggerObjectsOnPathStep(unit, row, col)` — на каждом шаге пути в `executeMove`. Сейчас обрабатывает `trap`.
- `triggerObjectsOnMoveEnd(unit, row, col)` — только на финальной фактической клетке. Сейчас обрабатывает `lure`.

На переходе к новой волне `state.objects = []`.

| kind | Источник | Срабатывание | Эффект |
|---|---|---|---|
| `trap` | Активный навык лучника `trap` (С22) | `triggerObjectsOnPathStep` — любой юнит шагнул на клетку, независимо от команды (свой/чужой/нейтральный). | Наносит `payload.dmg` физического урона + `applyImmobilized(2)`. Объект удаляется. Движение жертвы прерывается (canUnitMove видит свежий immobilized). |
| `lure` | Активный навык лучника `lure` (С22) | `triggerObjectsOnMoveEnd` — юнит ЗАКОНЧИЛ ход на клетке приманки (не на промежуточном шаге). | Объект удаляется. ## Aggro-система

NPC с полем `aggroState` (зомби сейчас) имеет два состояния:
- `'sleeping'` — не видит героев, ходит по `idleBehavior` (сейчас единственный — `wander`: случайная клетка в `moveRange`).
- `'active'` — действует по основной AI-политике (атака + движение к цели).

Метрика — Чебышев (`max(|dr|, |dc|)`). Сквозь стены/деревья НЕ блокируется (зомби «слышит/чует» сквозь препятствия — сознательный компромисс вместо полного LOS).

**Триггеры пробуждения `sleeping → active` (4):**

1. **После хода NPC** (idle wander): зомби сделал шаг — герой оказался в радиусе → проснулся к следующему своему ходу. Движение в этот ход не возвращается.
2. **После хода игрока** (`endTurn` → `checkAggroForAllNpcs`): герой подошёл к спящему врагу.
3. **При спавне новой волны** (`startNextWave` → `checkAggroForAllNpcs`).
4. **При получении урона от живого вражеского источника** (`applyDamage` → `wakeOnDamage`): закрывает дыру с дальнобойными атаками из-за пределов радиуса. DoT-тики (source=null) и friendly fire не будят.

Возврат `active → sleeping` отсутствует.

**Визуал:** бейдж `Z` (sleeping, серо-голубой) или `!` (active, красный пульсирующий) в правом верхнем углу спрайта NPC.

Hover на NPC даёт превью зоны — режим зависит от `aggroState`:
- **sleeping** → жёлто-оранжевая `.aggro-preview` заливка в Чебышевском `aggroRadius` (зона слежения, зайдёшь — заметит).
- **active** → красноватая `.move-preview` заливка по реальным клеткам, до которых NPC может дойти за один ход с текущей Скоростью (BFS через `computeReachableCells` с учётом деревьев и юнитов). Помогает игроку точно оценить угрозу и спланировать ход.

Превью пропадает в режиме прицеливания (`state.mode` задан) и при следующем `render()`.

## Порядок одного хода

Полный цикл от `beginTurn` до передачи хода следующему по инициативе (`core/turn.js`):

1. **`beginTurn`**: выбор активного юнита из `initiativeOrder` (мёртвые пропускаются), сброс `actionsUsedThisTurn` / `skillsUsedThisTurn` / `skipTurnThisTurn`.
2. **Глобальное +1 маны** (если ниже максимума).
3. **`expireTurnStartEffects`**: эффекты с `expiresAt:'turnStart'` снимаются ДО фазы хука.
4. **Фаза `onTurnStart`**: срабатывают хуки эффектов носителя. Если убили — отложенный `endTurn` через 280 мс (анимация смерти).
5. Если `skipTurnThisTurn === true` (Stun) — отложенный `endTurn` без передачи управления.
6. **Действия юнита.** Игрок (или ИИ) тратит движение / атаку / актив в любом порядке. Завершается явной командой «Конец хода» или ИИ.
7. **`endTurn`**: лог «завершает ход».
8. **Фаза `onTurnEnd`**: хуки эффектов носителя.
9. **`triggerPassivesAtTurnEnd`**: пассивки с `trigger:'onTurnEnd'` (сейчас `mana_regen`).
10. **`tickCooldowns`**: уменьшение `unit.cooldowns` на 1.
11. **`tickEffectsAtTurnEnd`**: уменьшение `remaining` эффектов на 1, снятие выдохшихся, переклэмп ресурсов (если убрался stat-debuff).
12. **`expireTurnEndEffects`**: эффекты с `expiresAt:'turnEnd'` снимаются здесь (Дальний выстрел и подобные).
13. **`checkAggroForAllNpcs`**: пробуждение спящих NPC, если герой подошёл в их радиус.
14. **`advanceTurn`**: продвинуть `turnIndex`. Закончился раунд — пересчёт `initiativeOrder`, `round++`.
15. **`checkVictory`**. Если бой не окончен — `beginTurn` для следующего юнита.

Авто-завершение хода **намеренно не делается**: игрок может думать, у некоторых классов остаются активы вне `actionsUsedThisTurn`. Конец хода — только явный (кнопка «Конец хода» или Enter).

## Волны и победа

- Игрок контролирует до 4 героев в низу (лимит = `HERO_SPAWN.length`). Враги спавнятся в случайных свободных клетках верхней половины (ряды 0–9).
- В начале волны случайно (`pickWaveGroup` в `core/state.js`) выбирается одна группа врагов из `WAVE_GROUPS`: `undead` (рядовые — `zombie` / `skeleton_warrior` / `skeleton_archer`, лидер — `ghost`) или `wolves` (рядовой — `wolf`, лидер — `wolf_alpha`).
- Размер волны фиксирован: `totalCount = 5` (правка 12.05.2026: рост массы по волне отменён — сложность теперь идёт через `personalLevel` и появление лидера).
- Если у группы есть лидер и `waveNumber >= leader.baseDifficulty` — добавляется ровно ОДИН лидер в зачёт count. Например: на 5-й волне `undead` это 4 рядовых нежити + 1 призрак (`ghost.baseDifficulty=5`); на 5-й волне `wolves` это 4 волка + 1 вожак.
- Все враги волны убиты → выжившие герои возвращаются на исходные позиции, восстанавливают полные HP/ману, эффекты и `actionsUsedThisTurn` сбрасываются. Надгробия от монстров чистятся. Мёртвые герои остаются мёртвыми (их клетки тоже чистятся).
- **Награда (С7-предметы 08.05.2026).** После победы `forceWaveVictory()` зовёт `generateRewardItem(missionDifficulty)` (Camp v1.5: difficulty берётся у целевого региона, а не из `state.wave.number`). Полученный инстанс кладётся в `state.partyInventory`. В лог пишется строка «Получена: <полное имя> (стоимость N)». На сложности 1 валидных комбинаций нет (минимальный дроп = cost 2), пишется системное сообщение «ничего не выпало (сложность N ниже минимума дропа)» — это согласовано с дизайном.
- **Camp v1.5 (09.05.2026): пост-миссионный поток.** После победы `forceWaveVictory` обновляет `state.regions` (target -3 clamp 1, others +1), запускает level-up очередь по выжившим участникам (state.units alive heroes); по завершении очереди (`advanceLevelUpQueue` пуст) — `endMissionCleanup` (тик `restingTurnsLeft` всех живых, установка =1 участникам, permadeath мёртвых, очистка `state.units`/`state.objects`) → `enterCampMain`. После провала (`forceMissionDefeat`, см. ниже) — все регионы +1 (без -3), `endMissionCleanup` без level-up; если в партии не осталось живых → `state.gameOver='defeat'`.
- Спавн следующей миссии — явный шаг игрока через UI лагеря (карточка региона на globalMap), не автоматический. `state.wave.number` теперь — cumulative счётчик миссий (для логов/статистики), а не «уровень волны».
- **Поражение в миссии** (все участники мертвы, но партия в целом ещё жива) → `forceMissionDefeat` → возврат в лагерь без награды и без level-up'а, с potential permadeath участников. Финальный `gameOver='defeat'` только при пустой партии (план — расширить до «и нет золота на найм» в Camp v3).

## ИИ зомби

Активный зомби (`aggroState === 'active'`) на свой ход (`AI_POLICIES.zombie` в `core/ai.js`):

**A. Атака на старте.**
1. Кого можно ударить с текущей клетки (манхэттен ≤ `weaponRangeOf` = 1 для Когтей).
2. Среди достижимых:
   - Тех, кого гарантированно убьёт одним ударом (`baseDmg ≥ hp`). Если несколько — мин. Удача → случайный.
   - Иначе среди всех — мин. HP → мин. Удача → случайный.
3. Атаковать.

**B. Движение.**
1. Цель для подхода: ближайший по BFS-пути герой (с учётом стен/могил/юнитов). Ничья — мин. HP → мин. Удача → случайный.
2. Двигаться по кратчайшему пути, тратя ровно `moveRange` клеток (или меньше).

**C. Атака после движения.**
- Если на шаге A атака не была потрачена — повторить проверку с новой клетки.

**Provoked-приоритет (Сессия 20).** Если на зомби висит `provoked` с живым `forcedTarget` в радиусе атаки — он атакует именно его, игнорируя обычные эвристики A. Если `forcedTarget` вне радиуса — стандартное движение к нему. После «вынужденного действия» (атака/шаг к forcedTarget) `consumeForcedMoveEffects` снимает provoked. Маскировка `forcedTarget` — НЕ исключение: provoked зафиксирован в прошлом ходу, и AI продолжает идти к цели независимо от текущей видимости.

**Camouflage-фильтр (Сессия 23).** Замаскированный герой исключается из выборки целей атаки (`aiAttackableTargets`) и движения (`heroes` в `aiZombieStepMove`) через единый предикат `isHiddenFromAI(unit, ai)`. Спящие зомби рядом с замаскированным героем НЕ просыпаются (`checkAggro` фильтрует `camouflage`). `wakeOnDamage` остаётся независимым каналом: если замаскированный успел нанести урон, жертва просыпается «от боли». AoE-навыки игрока ПОПАДАЮТ по замаскированному (зона определяется геометрией клетки), но камуфляж после такого попадания НЕ слетает — так выбран баланс «щит от прицельного выбора, не от взрывов».

**Lure-приоритет (Сессия 22).** Если на зомби нет provoked, но в радиусе вражеской приманки — `findAttractingLure` возвращает её, и AI идёт через `aiMoveTowardCell(u, lure.row, lure.col)` к клетке приманки (вместо обычного выбора героя). Атака приоритетнее: если в радиусе атаки есть подходящая цель, AI сначала атакует, потом идёт к лурe. Срабатывание lure (удаление + applyOnPickup) — на ЗАВЕРШЕНИИ хода жертвы на её клетке.

Спящий зомби (`aggroState === 'sleeping'`) идёт по `IDLE_POLICIES.wander`: случайная клетка через `computeReachableCells` → `executeMove` → `checkAggro` → `endTurn`. Не атакует, не использует активы. Lure действует и на спящих (запах работает независимо от состояния).

## ИИ волков

Волки и Волк-вожак используют общую функцию `wolfPolicy(u)` в `core/ai.js`. Отличие от зомби — приоритет цели атаки (учёт стаков «Совместной охоты»). Сбор стаи вокруг лидера убран балансной правкой 14.05.2026.

**Приоритет цели атаки** (`aiPickAttackTargetWolf`): из тех, кого волк может ударить с текущей клетки (манхэттен ≤ weapon.range):
1. Цели, которые волк гарантированно убивает одним ударом — оценка через `aiBaseDamageWolf` (учитывает `joint_hunt_marks` стаки на цели и `joint_hunt`-пассивку у атакующего).
2. Если таких нет — цели с **MAX** стаков `joint_hunt_marks` (если стаков нигде нет — пропускаем шаг).
3. Среди оставшихся — **MIN HP**.
4. Тай-брейк — мин. Удача → случайный.

**Приоритет цели движения** (`aiWolfStepMove`) — двухэтажный фильтр (правка 07.05.2026 по фидбэку):
1. Сначала отбираем врагов, до которых волк может **дойти И атаковать в этот же ход** (BFS-дистанция до соседней клетки цели ≤ `moveRange`). Если такой подсписок не пуст — приоритеты применяются ТОЛЬКО к нему: 1-shot → MAX стаков → MIN HP → MIN luk → random.
2. Если никого нельзя достать в этот ход — fallback к полному списку достижимых, тот же набор приоритетов; финальный тай-брейкер — БЛИЖАЙШИЙ по BFS (иначе волк бежал бы через всю карту к 1-HP цели мимо более близких).
3. Если на волке висит `provoked` — fallback не работает, идёт к forcedTarget безусловно.

**Активный волк и Волк-вожак (`AI_POLICIES.wolf` === `AI_POLICIES.wolf_alpha` === `wolfPolicy`):**
- A. Атака с места по wolf-приоритетам.
- B. Движение. Сначала `getForcedMoveDirective` (provoked / lure — те же приоритеты, что у зомби). Если ничего не форсируется — `aiWolfStepMove` к ближайшему врагу по wolf-приоритетам.
- C. Если атака на A не была потрачена — повторить с новой клетки.
- Финал: `consumeForcedMoveEffects` + `endTurn`.

**Спящие волки и вожак** идут по `IDLE_POLICIES.wander` — как зомби. Просыпаются от `checkAggro` (ближе `aggroRadius=6` Чебышев), `wakeOnDamage` или `wolf_howl` соседнего активного волка.

**Wolf Howl и инициатива.** В `triggerPassivesAtTurnStart` пассивка `wolf_howl` пробуждает спящих сородичей и (advanced/elite) вешает на них `wolf_howl_buff` (`expiresAt:'turnEnd'`, `statMods.spd:+2/+4`). Затем зовётся `refreshInitiativeAfterCurrent` (источник правды — `core/skills.js`): юниты, уже сходившие в этом раунде, остаются на местах; остальные пересортировываются по эффективным статам. Это позволяет ускорившимся волкам обогнать соседей по очереди в текущем раунде.

**Pack Leader аура.** В начале каждого хода (`beginTurn` ДО `expireTurnStartEffects`) зовётся `refreshPackLeaderAuras`: для каждого волка ищется живой лидер группы той же команды в радиусе Чебышева 5. Если есть — обеспечивается эффект `pack_leader_aura` с `statMods.str = floor(базовая Сила цели * 0.30)` (балансная правка 14.05.2026, раньше был `ceil`). Иначе — эффект снимается. Лаг между смертью лидера и снятием ауры с остатков стаи — один ход (приемлемо).

## Отладочный лог (DebugLog)

Встроенный инструмент для удалённой диагностики багов. Существует, потому что агент не может запустить игру в браузере сам — а воспроизводимый снимок состояния + лента последних событий обычно отвечают на 90% диагностических вопросов за один раунд.

### Как использовать (пользователь)

1. Воспроизвести баг.
2. Открыть DevTools (F12) → нажать кнопку **🛠** в правом нижнем углу поля.
3. В popover'е — кнопка **«📋 Дамп для агента»** (синяя). Кликнуть.
4. Появится alert «Скопировано в буфер обмена». Вставить (Ctrl+V) в чат.

### Что собирается

Один JSON, готовый для прямого парсинга:

- **timestamp + uptime_ms** — когда сделан снимок.
- **window** — innerWidth/Height, scrollX/Y, devicePixelRatio.
- **html / body** — scrollTop, scrollHeight, clientHeight/Width, computed `overflow`, inline-style.
- **viewport / battlefield** — getBoundingClientRect + текущие значения CSS-переменных `--cell` / `--pan-x` / `--pan-y`.
- **activeElement** — какой элемент сейчас в фокусе (tag/id/class).
- **state** — wave, mode, активный/выбранный юнит, gameOver, view (zoom/pan), activeLevelUp, levelUpQueue, число units/objects/trees.
- **VIEW** — текущие размеры viewport-камеры (width/height/cellPx/zoom-границы).
- **buffer** — последние 200 событий с временными метками.

### Хуки и категории

Хуки `DebugLog.log(category, msg, data)` встроены в:

| Категория | Где вызывается |
|---|---|
| `camera` | `applyView`, `setZoomAt`, `panBy`, `resetView` (`ui/zoom-pan.js`) |
| `wave` | `forceWaveVictory`, `startNextWave` (`core/state.js`) |
| `level-up` | `advanceLevelUpQueue`, `finishCurrentLevelUp` (`core/state.js`) |
| `action` | `dispatchActiveSkill` (`core/combat.js`) |
| `ai` | `runAiTurn` (`core/ai.js`) |
| `combat` | `applyDamage` (`core/damage.js`) |
| `render` | `render()` top-level (`render/render.js`) |
| `scroll` | window scroll event (auto-перехватчик в `debug-log.js`) |
| `dom-mutation` | MutationObserver на inline-style html/body (auto) |
| `error` | window error / unhandledrejection (auto) |

### Из консоли

```js
DebugLog.snapshot()        // объект для inspect
DebugLog.dump()            // JSON-строка
DebugLog.copyToClipboard() // в буфер обмена
DebugLog.disable('render') // отключить шумную категорию
DebugLog.enable('render')  // включить обратно
DebugLog.clear()           // очистить буфер
DebugLog.buffer            // массив записей (read-only copy)
```

### Что НЕ собирается

Сознательно не пишем в снапшот:

- Полный массив `state.units` (тяжёлый; обычно достаточно `unitsCount`). Если для конкретного бага нужно — добавить ad-hoc в snapshot из консоли: `JSON.stringify({...DebugLog.snapshot(), units: state.units}, null, 2)`.
- `innerHTML` крупных контейнеров (overlay, panels) — берётся из DevTools при необходимости.
- Ссылки на DOM-узлы (не сериализуются).

### Удаление перед релизом

Один файл + один script-тег. Чтобы вырезать:

## Как пополнять этот документ

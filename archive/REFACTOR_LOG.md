# Рефакторинг архитектуры — план и прогресс

> Этот файл — оперативный документ рефакторинга. Карта файлов, статусы сессий,
> правила добавления нового контента после рефакторинга, лог регрессий.
> При возвращении к проекту в новой сессии — читать **после** `DESIGN.md` и `CODEX.md`,
> если работаем над рефакторингом или над фичей, которая трогает несколько слоёв.

## Зачем рефакторинг

Прототип `index.html` к моменту старта рефакторинга — монолит ~3500+ строк со встроенными
CSS, HTML и JS. Стоимость каждой даже мелкой правки растёт нелинейно: агенту приходится
сканировать весь файл, чтобы найти нужное место и убедиться, что соседний код не сломается.
Игра планируется как долгоиграющий проект с регулярными расширениями (новые классы, монстры,
навыки, режимы, UI-блоки). Цель рефакторинга — сделать так, чтобы 95% правок укладывались
в 1–2 файла, без чтения остального.

См. также соответствующий раздел в `DESIGN.md` («План: рефакторинг архитектуры файлов»).

## Решение по языку

**Остаёмся на ванильном HTML/CSS/JS.** Используем **обычные** `<script src="...">` теги
(НЕ `type="module"`) — они работают из `file://` во всех браузерах, а ES-модули в Chrome
из `file://` **заблокированы** политикой безопасности (null-origin). Двойной клик по
`index.html` должен продолжать работать без dev-сервера и без флагов.

**Что мы теряем без модулей:** синтаксис `import`/`export`. Файлы общаются через
глобальный scope — это ровно как работает текущий монолит (все функции/константы
в `<script>`-теге уже глобальные). Семантической регрессии нет.

**Что мы приобретаем:** разделение по доменным файлам, маленькие файлы (≤300 строк),
быстрая навигация, возможность открывать ровно 1–2 файла на правку.

TypeScript / другие языки рассматривать сейчас не будем. Если когда-нибудь понадобятся типы —
добавим JSDoc с `// @ts-check` отдельным проходом, без переписывания кода.

### История: попытка R3 (промежуточный `main.js`) отвергнута

Изначально план предполагал R3 — вынести **весь** JS из `<script>` в один большой
`src/main.js` как промежуточный шаг, а уже потом дробить его. На практике это оказалось
непрактично:

- **Токеновый бюджет.** JS-блок ~3155 строк, ~60k токенов. Прочитать ещё можно (4 захода
  по Read), но сгенерировать те же 60k в одном Write для `src/main.js` плюс удалить из
  `index.html` через Edit — это удвоение нагрузки без полезной выгоды.
- **FUSE-mount в песочнице** в моменты, когда файл активно редактируется через Edit, может
  показывать bash-у устаревший снимок (наблюдалось в начале R3). Это блокирует использование
  `sed` для крупной хирургии.
- **R3 не имеет самостоятельной ценности** — это был просто перенос кода, не дающий
  никакой модульности. Цель рефакторинга достигается прямым извлечением мелких блоков.

**Новый подход:** inline `<script>...</script>` в `index.html` остаётся «storage»-ом
исходного кода. Каждая последующая сессия извлекает **один логический блок** в свой
файл (`src/data/*.js` или `src/core/*.js`), подключает его через `<script src="...">`
**ПЕРЕД** inline-блоком — так что глобальные определения видны inline-коду естественным
образом. К финальной сессии inline почти опустеет, и оставшийся «оркестрационный»
код переедет в `src/main.js` или удалится, если станет не нужен.

## Принципы

1. **Один файл — одна узкая ответственность.** Цель — 95% правок укладываются в 1–2 файла.
2. **Данные отдельно от кода.** Все справочники (`CLASSES`, `WEAPONS`, `SKILLS`, иконки) — в `src/data/`.
3. **CSS по доменам, не по типам.** `units.css`, `tombstone.css`, `panel.css` — а не «все цвета в одном».
4. **Каждая сессия оставляет прототип играбельным.** Это инвариант. Регрессия → откат к бекапу до сессии.
5. **Карта файлов в `CODEX.md` и в этом файле** (см. ниже) — главный навигатор. Открываешь карту,
   находишь нужную фичу, открываешь один файл.
6. **Логика боя НЕ переписывается.** Только разнос по файлам. Формулы, эффекты, анимации,
   хоткеи — идентичны до и после.
7. **Глобальная проверка только при сбое тестов или явных багах.** В обычной правке
   агент не перечитывает соседние файлы — доверяет карте и контракту функций.

## Политика бекапов (для рефакторинга)

Перед каждой сессией R* — пользователь делает бекап вручную через Проводник в
`C:\Проекты\tactics\backup\<YYMMDD-HHMM>\`.

**Что копировать:**
- `index.html`
- `DESIGN.md`
- `CODEX.md`
- `REFACTOR.md`
- папку `assets/` целиком
- папку `styles/` целиком (после R1)
- папку `src/` целиком (после R3)

**Что НЕ копировать:**
- папку `backup/` (бекап бекапов — бессмысленно)

**Простое правило:** копируй всё содержимое `C:\Проекты\tactics\` **кроме** папки `backup/`.

## Регрессионная проверка

Каждая сессия R* заканчивается **обязательным игровым тестом**, прежде чем перейти к следующей.

**Минимальный сценарий проверки:**
1. Открыть `index.html` двойным кликом — страница грузится без ошибок в DevTools Console.
2. Стартует бой, видны три героя внизу и зомби сверху.
3. Завершить ход первым героем (Enter) — переход хода работает.
4. Атаковать ближайшего зомби (A → клик) — урон в логе совпадает с ожидаемым.
5. Если есть маг — кастануть фаербол (F → клик) — вспышка анимации, урон по AoE.
6. Дойти до победы над волной — переход на следующую волну работает, герои
   возвращаются на старт, HP/мана восстанавливаются.
7. Зум колесом, пан правой кнопкой/стрелками — работают.
8. Hover на надгробие — тултип показывается.

Если что-то из этого ломается — **остановка**, разбор причины. Если за разумное время не
чинится — откат к бекапу до текущей сессии, документирование проблемы в этом файле
(раздел «Лог регрессий»), пересмотр плана сессии.

поведение CSS при подключении через `<link>` идентично инлайн-стилям, разница только
в том, что брезер делает ещё один HTTP-запрос (для `file://`-открытия — мгновенный).

**Действия:**
- Считан весь блок `<style>...</style>` из `index.html` (строки 26–945, итого 920 строк
  включая теги; 918 строк собственно CSS).
- Создан `styles/all.css` (928 строк = 918 оригинальных + 10 строк header-комментария).
  Форматирование сохранено как было (включая ведущие 2-пробельные отступы из `<style>`-блока) —
  никаких косметических изменений, только перенос. Нормализация отступов отложена
  на потенциальный следующий проход (если когда-нибудь понадобится).

### R13 — Извлечь движение и BFS-инфраструктуру в `src/core/movement.js`
**Цель:** забрать из монолита всё, что про «грид и движение по нему» —

**Что переехало (в порядке появления в новом файле):**

**Тонкость с keydown.** В системе теперь ДВА window keydown-handler-а:
один — внутри `setupViewInteractions` для камера-клавиш (Arrow/+/-/0),
второй — `bindHotkeys` для действий (Esc/M/A/F/Enter). Они не
пересекаются по ключам, оба используют `e.preventDefault()`. Если
один обработчик не обработал ключ (`handled = false`), событие
распространяется ко второму. preventDefault внутри одного не мешает
другому — каждый отвечает только за «свои» ключи.

**Тонкость с упрощением init().** Старый init() имел 60 строк с тремя
inline-обработчиками; новый init() — 14 строк, шесть императивных
вызовов (createInitialState → startNextWave → 3×bindXxx →
setupViewInteractions → render → applyView). Точка ответственности
ясна: что-то не работает в кликах поля? — `ui/input.js`. В кнопках
панели? — `ui/input.js`. В хоткеях? — `ui/hotkeys.js`. В камере? —
`ui/zoom-pan.js`. До R18 всё это было в одной длинной функции.

**Тонкость с порядком загрузки.** ui-файлы подключаются ПОСЛЕДНИМИ
(после `render/*`, перед inline). Безопасно потому, что:
1) ui-функции вызываются только из тел других функций (init,
   render-overlay, render-units и т.п.), резолв при вызове;
2) `bindXxx` дёргаются из `init()` (inline), к этому моменту все
   25 внешних script-ов выполнены и DOM готов;
3) `enterMode`/`exitMode` зовутся из render-overlay (mouseenter
   handlers внутри `renderOverlay`), но render-overlay вызывается
   ПОСЛЕ init-bind (зачем? потому что все клики проходят через
   bind-функции, которые включают render() в свою цепочку).

**Метод правки.** Один атомарный Python-скрипт обновил два файла
(index.html + state.js) транзакционно: сначала все assert-ы границ
и уникальности, потом две атомарные write_text-операции.
REFACTOR.md обновлён таким же атомарным Python-скриптом
(`r18_refactor_md.py`).

**Особый момент с `*/` в комментариях.** В первой версии ui-файлов
в шапке был фрагмент `Arrow*/+/-/0` (попытка кратко описать
горячие клавиши). Звёздочка-слэш закрывала JS-комментарий
прямо посреди фразы, и `node --check` падал с
`SyntaxError: Invalid regular expression flags`. Починено заменой
на `Arrow-клавиши/+/-/0` во всех трёх файлах. Урок: в JS
многострочных комментариях избегать последовательности `*/`,
даже если она логически часть текста.
